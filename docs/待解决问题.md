# WeCom Agent 待解决问题

更新于 2026-07-03。

## 当前结论

本仓库不能再写成“完全砍掉自建应用”。更准确的策略是：

- B 线本地解密读取继续作为默认读取方案。
- A 线主动操作优先复用 WecomTeam `wecom-cli/wecom-unified`。
- 如果企业不开放授权机器人对应权限，A 线缺口需要两种 fallback：企业管理员开权限，或恢复最小自建应用 API。
- 实时互动优先接 WecomTeam Bot SDK；旧 HTTP 回调代码只作为 legacy 参考。

## 建议开发路线

### Phase 1：先把 A 线边界收稳

1. 补 `wecom-cli` 自检脚本。
2. 明确记录当前企业支持 `todo`，不支持 `contact/msg/schedule/meeting`。
3. 用户确认后跑 `todo create/delete` 与 `doc create/edit` 写操作测试。

### Phase 2：决定 fallback

1. 如果管理员能开放授权机器人权限，继续走 WecomTeam CLI。
2. 如果不能开放，恢复最小自建应用 API fallback：
   - 发消息
   - 通讯录
   - 日程
   - 会议
3. fallback 只补 A 线缺口，不参与本地聊天读取。

### Phase 3：接 Bot WebSocket

1. 用 `wecom-aibot-python-sdk` 做最小监听和回复。
2. 支持认证、心跳、重连、文本消息、流式回复。
3. 后续补图片/文件下载解密和主动推送。

### Phase 4：拆分 skill 文档

建议结构：

```text
references/
  local-decrypt.md
  wecom-cli.md
  bot-sdk.md
  legacy-self-built-app.md
  safety.md
```

`SKILL.md` 只保留路由规则和写操作确认原则。

## 不再重复讨论的边界

- 历史聊天读取默认走 B 线本地解密，不走 `wecom-cli msg get_message`。
- WecomTeam CLI 能力受企业授权机器人权限限制，不是所有企业都完整支持。
- 自建应用不是默认主线，但在当前企业环境下可能仍是完整 A 线 fallback。
- 任何发消息、建文档、建待办、删改日程会议都必须先确认对象、内容和影响范围。
