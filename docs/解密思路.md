# 企业微信(macOS)消息库解密思路

> 环境：企业微信 5.0.8 / macOS Apple Silicon。最终验证成功，从内存提取出解密的消息库 SQLite 页。

## 核心思想

**不去"破解密钥解密文件"，而是"偷取程序已经解密好的成品"。**

企业微信运行时必然要把加密的消息库解密成明文供 SQLite 读取——只要能进入进程内存，把这些解密后的页捞出来，加密就形同虚设。难点只剩两个：**进得去**（绕过所有防护注入）+ **找得到**（在内存里定位解密页）。

---

## 为什么"正面破解密钥"走不通（已逐一证伪）

| 尝试 | 结果 | 原因 |
|---|---|---|
| 拿 CorpID+Secret 走官方 API 读消息 | ✗ | 官方不开放读聊天，需会话存档(盖章+付费) |
| 套用个人微信的 SQLCipher 参数 | ✗ | 企微是 wxSQLite3 + 自定义方案，参数不同 |
| frida hook 抓 key | ✗ | 登录时反调试检测 frida → 崩溃 |
| hook 所有导出加密函数(EVP/AES/CBC/aes_v8) | ✗ dump=0 | **消息库用静态编译的 AES，无导出符号，hook 不到** |
| 1023 个捕获 key × 全模式/IV/reserve/PBKDF2 | ✗ 全 MISS | 消息库 key 不在导出路径里 |

**关键认知**：消息库的加密走 WCDB 静态链接的私有 AES，外部无法 hook，正面拿 key 这条路根本不通。

## 成功路线：注入 + 内存 carve（5 步）

### ① 重签名（绕过 hardened runtime + library validation）
```bash
# 从原始 dmg 提取完整 entitlements（保留 application-groups，否则登录崩）
codesign -d --entitlements :- <原始app> > ent.plist
# 加两个关键 entitlement
/usr/libexec/PlistBuddy -c "Add :com.apple.security.get-task-allow bool true" ent.plist
/usr/libexec/PlistBuddy -c "Add :com.apple.security.cs.disable-library-validation bool true" ent.plist
# 重签 —— 【关键】绝不用 --deep
codesign --force --sign - --entitlements ent.plist /Applications/企业微信.app
```
> **最大的坑**：`--deep` 会把主程序的 entitlements 灌给所有 helper，破坏它们 → 登录失败/秒退。不加 `--deep` 只签主 bundle，helper 保留原签名 → **会话保留、auto-login 正常**。

### ② DYLD 注入自写 dylib
```bash
open /Applications/企业微信.app --env "DYLD_INSERT_LIBRARIES=$PWD/keyhook.dylib"
```
- hardened runtime 已去 → DYLD 注入生效
- disable-library-validation → 第三方 dylib 能加载
- 非 frida → 反调试检测不到（frida 在登录时必被杀，dylib 不会）
- dylib 日志/输出要写**沙盒容器内**(`~/Library/Containers/com.tencent.WeWorkMac/Data/`)，写 /tmp 被沙盒拒

### ③ auto-login（不触发 QR 超时）
- 有保存的会话 → 注入启动后几秒自动登录，无需扫码
- 无会话则停在 QR，约 13s 超时退出（窗口太短）→ 所以必须先有会话

### ④ dylib 内起线程，mach_vm 扫自身内存找解密 SQLite 页
```c
// 在进程内用 mach_vm_region + mach_vm_read_overwrite 遍历 rw 内存
// 识别 SQLite b-tree 页: 首字节 0x0d/0x05/0x0a/0x02 + cell数/cellstart 合理
```
> **第二大坑 / 关键修正**：SQLite pager 缓存的页是 **malloc 分配的，不是 4096 对齐**。
> - 4096 对齐粗扫 → 只抓到恰好 mmap 对齐的 CEF 浏览器缓存页（cfurl_cache 等），**抓不到 WeCom 消息页**
> - 改 **512 字节步长细粒度扫描** → 立刻抓到 malloc 的消息库页

### ⑤ 解析 carve 出的明文 SQLite 页
- carve 到 320 页解密页，含完整表结构：`message_table` / `history_message_section_table` / `conversation_tab_info_table_v9` / `wechat_contactV1` 等
- 自写 b-tree leaf 页解析器（varint + record header + serial types）提取结构化记录
- 提取出真实会话、联系人、消息正文（"该消息已被撤回"、"ok 感谢"、时间戳等）

## 关键技术栈
- `codesign` —— 重签名（不用 --deep 是精髓）
- `DYLD_INSERT_LIBRARIES` —— 注入（替代 frida，避反调试）
- ARM64 inline hook —— 前 16 字节非 PC 相对指令的函数可 patch（trampoline + 绝对跳转）
- `mach_vm_region` / `mach_vm_read_overwrite` —— 进程内只读内存扫描
- SQLite b-tree 页格式 —— 解析明文页提记录

## 双层加密说明
企微消息有两层：
1. **DB 层**：整个 Info.db / message 库用 wxSQLite3 加密（本方案 carve 破的是这层）
2. **消息层**：部分消息正文再单独 AES-256-GCM（即捕获到的 NID 901 key）

内存 carve 拿到的是**两层都解完的成品页**，所以直接可读。
