# 企业微信(macOS)消息库解密 —— 实际方案（已实跑验证）

> 环境：企业微信 5.0.8 / macOS Apple Silicon。已导出 5458 条结构化消息（会话/发送者/时间/正文）。
> 本文是**真方案**。早期"内存 carve 解密页"那套是**误判**，见文末"弯路"。

## 一句话

消息库**不是 SQLCipher**，是 **wxSQLite3 AES-128-CBC**（plaintext-header 变体）。16 字节 key **只在登录后的进程内存里**。
**活进程内存扫出 key → 离线按 wxSQLite3 算法解密整库 → 解析 `MESSAGE` 表**。

## 方案识别（最关键，先认对家族）

`Messages1/Info.db` 头：
- offset 0–7：恒定 magic `a4 1d 91 37 59 95 d7 d6`（= **该库 key 的指纹**，同 key 的库相同；**不是**随机 salt）。
- offset 8–15：变化（wxSQLite3 把原 SQLite 头 16–23 搬到这）。
- offset 16 起：**明文** SQLite 头（`10 00`=页4096…）→ 用了 `cipher_plaintext_header_size`。

→ 这是 wxSQLite3 plaintext-header AES-128。**SQLCipher 的头 16 字节是随机 salt 且每库不同**，对不上 → 排除 SQLCipher（早期 1023 个 key 全 MISS 就是因为一直按 SQLCipher 试解，家族错了）。

**算法参数**：AES-128-CBC；页 key = `md5(rawkey[16] + 页号<u32 LE> + b"sAlT")`；IV = SQLite3MultipleCiphers `generate_initial_vector(页号)`（LCG→md5）；无 HMAC；reserve=0；页1 解密前把 offset 8–15 还原回 16–23。实现见 `decrypt/wxwork_crypto.py`（含离线自测）。算法参考 `ylytdeng/wechat-decrypt`。

## 四步

### 1. 重签名（为了能读进程内存）
adhoc 重签（**绝不用 `--deep`**）+ 原始 entitlements + `get-task-allow` + `disable-library-validation`。保证 `task_for_pid` 可读内存、企微仍 auto-login。坑见 `decrypt/NOTES.md`、[[wechat_codesign_pitfall]]。

### 2. 活进程内存扫 16 字节 key（难点）
- key 只在登录开库时设入静态 WCDB codec，**只在内存**（离线 allkeys.txt / carve.bin 里都没有；内存里也**非 hex 字面量**，正则扫不到）。
- `task_for_pid` + `mach_vm_read` **只读**扫内存（不注入），每个 16B 窗口做**单块校验**：`md5` 派生页 key → 1 次 AES 解页1首块 → 比对明文片段。纯 Python 太慢 → 用 **CommonCrypto 写的 C 校验器 `validate.dylib`**（原生速度，AES-NI）。
- 实测扫到 ~5.7GB 命中：messages/session 一个 key、contact 另一个。
- 脚本：`decrypt/find_key_fast.py`（+`validate.c`）。**前提：企微运行并登录**（key 才在内存）。离线兜底 `find_key_offline.py`（试 allkeys.txt/carve.bin，通常不中）。

### 3. 离线全库解密
`decrypt/decrypt_wxwork.py`：用 key 按 wxSQLite3 算法解密 profile 下所有加密 `.db`（实测 53 个全成功）→ `decrypt/decrypted/`。明文库直接拷贝。**此步不需要企微开着、不需要重签**——纯文件+key。

### 4. 结构化解析
`decrypt/export_wxwork.py`：解析 `Messages1/Info.db` 的 **`MESSAGE`** 表
（`conv_id` / `sender_id` / `content_type` / `send_time`(unix秒) / `content`(protobuf BLOB)）→ CSV/JSON。
- 正文：protobuf 递归抽最内层文本（注意框架字节）。
- 会话名：`CONVERSATION.name`；发送者名：`Session.db` 的 `USER`(RID→name)。
- `SubMessageDB` 为空 = 无分库，全在 Info.db。

## 弯路（避免重走）

| 试过 | 结果 | 真因 |
|---|---|---|
| 按 SQLCipher 参数试 1023 个 key | 全 MISS | 家族就错了，是 wxSQLite3 |
| 内存 carve "解密好的 b-tree 页" | 只捞到 CEF/系统页 | 消息页不留标准 pager 缓存；且根本不需要 carve 页 |
| frida hook 抓 key | 登录被反调试杀 | 用 DYLD 注入/只读内存替代 |
| hook 导出的 OpenSSL(EVP/AES) | dump=0 | 消息库走静态链接 AES，无导出符号 |
| 以为需二进制逆向 WCDB AES | 不必要 | 内存找 key + 离线 wxSQLite3 解密即可 |

## 局限 / 下一步

- key 可能随**重新登录/换号**变化 → 失效时重跑 `find_key_fast.py`。
- **WAL**：最新几条可能在 `.db-wal` 未合并进主库，有秒级延迟（监控场景需读 wal 或接受延迟）。
- **消息类型**：38 种 `content_type`，文本(0/2)已通；媒体(语音14/15、图片29、文件)、卡片(145/565…)、系统事件(38/501/1011)的精细解析在优化中。
- 增量监控（按 max LID 取新消息）、发送者/群成员姓名解析在做。
