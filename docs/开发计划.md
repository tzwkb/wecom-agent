## B 线：本地读取/解析 —— 当前开发重点

链路已通：`find_key_fast.py`(扫key) → `decrypt_wxwork.py`(解密) → `export_wxwork.py`(导出)。剩**解析精度**：

- [ ] **B1. 发送者姓名**：`sender_id` → `Session.db` `USER`(RID→name) 解析为真名（现在还是数字 id）
- [ ] **B2. 消息类型分类**：`MESSAGE.content_type` 共 38 种——文本(0/2)已通；需给媒体(语音14/15、图片29、文件)、卡片(145/565/573…)、系统事件(38/501/1011)、会议(1073)、文档链接(13) 标注类型并合理呈现（媒体显示 `[语音]/[图片]/[文件]` + 必要时提 URL）
- [ ] **B3. 群成员名**：群聊里 `sender_id` 可能不在 USER 表 → 解析 `CONV_MEMBER_DATA.member_bin`(protobuf) 或其他来源
- [ ] **B4. 增量监控 `monitor.py`**：用 key 定时解密 Info.db，按 max `LID`/`send_time` 取**增量**新消息（无需重扫内存/重签，纯文件+key）。注意 WAL 延迟
- [ ] **B5. 媒体导出**：图片/文件/语音的定位与解密导出（语音可接转写）
- [ ] **B6. 一键封装**：`find_key_fast → decrypt → export` 串成一条命令；key 已存则跳过内存扫描
- [ ] **B7. Windows 端**：当前仅 macOS（社区有 wxwork 脚本可参考）

风险：① key 随重新登录/换号可能变→失效重跑 find_key_fast；② 解密读取需对企微 adhoc 重签（破坏性，日常用需重装恢复，见 [[wechat_codesign_pitfall]]）。
